# Analysis jobs

The core data-plane. Submitting a job queues work for the `AnalysisWorker` background service; results become available when the job's `status` reaches `completed`.

| Endpoint | Verb | Auth |
|---|---|---|
| `…/analysis` | `POST` | required |
| `…/analysis` | `GET` | required |
| `…/analysis/{id}` | `GET` | required |
| `…/analysis/{id}/results` | `GET` | required |
| `…/analysis/{id}/diagnostics` | `GET` | required |

(Report and dataset retrieval live on their own pages: **[Reports](reports.md)**, **[Datasets](datasets.md)**.)

## `POST /analysis`

Submit a new analysis job. Returns immediately — the worker channel picks it up asynchronously.

### Request

```json
{
  "uploadId": "…",
  "packs": ["windows-core@1.0.0", "iis-core@1.0.0"],
  "includeDataset": true,
  "selectedBaselineId": "…"
}
```

| Field | Required | Notes |
|---|---|---|
| `uploadId` | yes | From a prior [upload](uploads.md). |
| `packs` | yes | At least one pack reference. Use `<packId>@<version>` (explicit) or `<packId>` (auto-resolves to latest). |
| `includeDataset` | no | If `true`, the analyzer also persists a gzipped dataset artifact retrievable via the dataset endpoint. Default `false`. |
| `selectedBaselineId` | no | If set, the worker also runs `IAutoCompareService` against this baseline on completion. The baseline must be a `completed` job. |

### Responses

- `202 Accepted` with `Location: …/analysis/{jobId}` and body `{ analysisId, status: "queued" }`.
- `400 Bad Request` if `packs` is empty, or `selectedBaselineId` is set but the baseline isn't completed.
- `404 Not Found` if `uploadId` or `selectedBaselineId` don't exist.

### Example

```bash
WS=00000000-0000-0000-0000-000000000002
curl -X POST http://localhost:5043/api/workspaces/$WS/analysis \
  -H "Authorization: Bearer pal_xxx" \
  -H "Content-Type: application/json" \
  -d '{"uploadId":"…","packs":["windows-core"]}'
```

## `GET /analysis`

List jobs in the workspace, optionally filtered by status.

### Query

| Param | Notes |
|---|---|
| `status` | Filter — typical values `queued`, `running`, `completed`, `failed`. |

### Response

```json
{
  "items": [
    { "id": "…", "status": "completed", "uploadId": "…", "packs": [...], "createdAt": "…" }
  ]
}
```

## `GET /analysis/{id}`

Get a single job's record — status, timestamps, packs, baseline metadata.

### Response

```json
{
  "id": "…",
  "status": "completed",
  "uploadId": "…",
  "packs": ["windows-core@1.0.0"],
  "isBaseline": false,
  "baselineType": null,
  "baselineContextJson": null,
  "baselineLabel": null,
  "createdAt": "…",
  "completedAt": "…",
  "errorMessage": null
}
```

Returns `404` if the job doesn't exist (or isn't visible to this tenant).

## `GET /analysis/{id}/results`

Returns the report summary embedded in the job record — typically the `summary`, finding counts, and the top findings without their full evidence blobs. For the full document, use **[`GET /analysis/{id}/report`](reports.md)**.

### Status codes

- `200 OK` — results returned.
- `404 Not Found` — job unknown.
- `409 Conflict` — job is not yet `completed`.

## `GET /analysis/{id}/diagnostics`

Returns guided diagnostic insights generated by `IDiagnosticsService`. Insights are rule-based, derived from findings, worsening/appearing trends, and both-worsening correlation pairs — every insight cites the `affectedRuleIds` that drove it. No black-box inference.

### Response

```json
{
  "items": [
    {
      "id": "…",
      "title": "CPU saturation co-occurs with disk latency",
      "category": "cpu",
      "severity": "warning",
      "affectedRuleIds": ["high-cpu-sustained", "sustained-disk-read-latency"],
      "sourceDirection": "correlation"
    }
  ]
}
```

### Status codes

- `200 OK` — even when there are no insights (`items: []`).
- `404 Not Found` — job unknown.
- `409 Conflict` — job not yet `completed`.
