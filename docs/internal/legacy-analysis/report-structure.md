# PAL v2 — Report Structure and Grouping

Source: `legacy/pal-v2/PAL2/PALWizard/bin/Debug/PAL.ps1`, `GenerateHtml` function (line 4789) and `SaveXmlReport` function (line 5456).

---

## Output Formats

PAL generates three optional output formats from the same analysis run:
- **HTML** — Primary human-readable report (controlled by `$IsOutputHtml`)
- **XML** — Machine-readable version of all alerts and counter stats
- **CSV** — Raw counter values (pre-existing intermediate format from `relog.exe` conversion)

---

## HTML Report Sections (in order)

### 1. Header

Black banner containing:
- Report title: `Analysis of "<log filename>"`
- Report generation timestamp (localized datetime format)
- "PALv2" brand link

### 2. On This Page (Table of Contents)

Two-level navigation list:
- **Alerts by Chronological Order** — sub-items for each quantized time slice, with alert counts
  - Format: `[time range] ([W] warnings, [C] criticals)` — only slices with alerts appear as sub-items
- **Category groups** — one top-level entry per category (e.g., `Process`, `Memory`, `SQLServer:Buffer Manager`)
  - Each category lists its active analyses with alert counts
  - Analyses with no thresholds (stats-only) appear with `(Stats only)` suffix
- **Incomplete analyses** — link to the index of analyses skipped due to missing counters
- **Disclaimer**

### 3. Tool Parameters

Key-value table listing the analysis run configuration:

| Parameter            | Description                                      |
|----------------------|--------------------------------------------------|
| Log Time Range       | Start to end timestamp of the counter log        |
| Log(s)               | File path(s) of the input log                    |
| AnalysisInterval     | `AUTO` or human-readable time window             |
| Threshold File       | Path to the XML threshold file used              |
| AllCounterStats      | Whether overall stats were calculated            |
| NumberOfThreads      | Parallel execution thread count                  |
| IsLowPriority        | Whether analysis ran at low CPU priority         |
| DisplayReport        | Whether report was auto-opened after generation  |
| Script Execution Duration | Wall-clock time to run the full analysis    |
| Question Variables   | Each user-answered variable (e.g., `OS`, `PhysicalMemory`) |

### 4. Alerts by Chronological Order

Central summary table showing all alerts across all analyses ordered by time.

**Table structure per time slice:**

```
Time Range | Condition | Counter | Min | Avg | Max | Hourly Trend
```

- Each value cell is background-colored with the threshold color that the value violated (Yellow for Warning, Red for Critical, White for not violated).
- White text is applied over red/yellow backgrounds for contrast.
- The Condition cell links to the analysis section below.
- Only non-`ISINTERNALONLY` alerts are shown.

If no alerts were found across the entire log, a `No Alerts Found` row replaces the table.

### 5. Category Sections (repeating)

Each category (`Process`, `Memory`, `Network Interface`, `SQLServer:Buffer Manager`, etc.) becomes a top-level `<H1>` section. Within each category, enabled analyses whose counters were all found in the log are rendered in `<H2>` subsections.

**Per-analysis sub-structure:**

#### a. Analysis Header
`<H2>` with anchor named after the analysis (URL-safe). Enables the TOC links.

#### b. Description Block
If the `<DESCRIPTION>` element contains non-empty HTML, it is rendered as a block-quoted section preceded by `Description:`.

Typical description content (from XML source):
1. What the counter measures (technical definition)
2. Why high/low values are significant
3. Threshold values restated in plain language (Yellow: ..., Red: ...)
4. Next Steps — diagnostic actions to take when the threshold fires
5. References — links to Microsoft documentation, blog posts, KB articles

#### c. Chart Images
Each `<CHART>` element in the analysis produces an embedded `<IMG>` tag. Charts are PNG files pre-generated by `PAL.ps1` using `System.Windows.Forms.DataVisualization.Charting`.

- Charts with thresholds (`ISTHRESHOLDSADDED="True"`) display a yellow band for Warning range and red/tomato band for Critical range.
- The `ALT` text on each image includes the chart title and the warning/critical ranges in human-readable form (e.g., `Warning Range: 300 to 640`).
- If no data was available for a chart, a `No data to chart` placeholder table is rendered instead.

**Chart band direction:**
- `BACKGRADIENTSTYLE="BottomTop"` — used for "should stay above" counters (Page Life Expectancy, Available MBytes, Buffer Cache Hit Ratio). The band occupies the **bad zone below** the threshold.
- Default (`TopBottom`) — used for "should stay below" counters. The band occupies the **bad zone above** the threshold.

#### d. Overall Counter Instance Statistics Table

Below the charts, a table shows aggregate statistics for each counter instance across the full log.

**Table columns:**

| Column                   | Description                                                  |
|--------------------------|--------------------------------------------------------------|
| Condition                | OK (green), Warning (yellow), Critical (red), or unchecked  |
| Counter path             | Full `\\computer\Object(Instance)\Counter` path             |
| Min                      | Minimum value in the entire log                              |
| Avg                      | Average value in the entire log                              |
| Max                      | Maximum value in the entire log                              |
| Hourly Trend             | Calculated hourly slope (positive = increasing over time)    |
| Std Deviation            | Standard deviation of all values                             |
| 10% of Outliers Removed  | Average with the outermost 10% of values excluded            |
| 20% of Outliers Removed  | Average with the outermost 20% of values excluded            |
| 30% of Outliers Removed  | Average with the outermost 30% of values excluded            |

The outlier-removed averages exist to reveal the "typical" average when a small number of extreme spikes would otherwise skew the mean.

The `Condition` cell color comes from an `INTERNAL_OVERALL_COUNTER_STATS_` alert if one was generated.

### 6. Incomplete Analyses

An index (`<A NAME="IndexOfJobs">`) listing all analyses that were skipped because one or more required counters were not found in the input log. This helps the user understand which threshold rules did not run and why.

### 7. Disclaimer

Standard footer noting that PAL thresholds are guidelines, not definitive statements, and results require context-specific interpretation.

## Quantized Time Model

PAL divides the log duration into a fixed number of **time slices** (`$alQuantizedTime`). The default is 30 slices across the full log, regardless of log length.

- Each slice has a start/end timestamp.
- Counter values are pre-aggregated into `QuantizedMin`, `QuantizedAvg`, `QuantizedMax`, and `QuantizedTrend` arrays, one entry per slice.
- Alerts carry a `TIMESLICEINDEX` integer (0-based) linking them back to their slice.
- The `GetQuantizedTimeSliceTimeRange` function converts an index to a human-readable time range string.

This model makes a 1-hour log and a 7-day log both produce 30 "bins" in the report, keeping the chronological alert table a manageable size regardless of data length.

## Category Ordering

Categories are drawn from the `CATEGORY` attribute on each `<ANALYSIS>` element. PAL maintains a `$global:alCategoryList` (ordered list) that determines the rendering sequence. Categories appear in the report in the order they are first encountered while loading the threshold file.

Common category names (from XML source):
- `Process`
- `Memory`
- `Processor`
- `Network Interface`
- `PhysicalDisk`
- `SQLServer:Access Methods`
- `SQLServer:Buffer Manager`
- `SQLServer:SQL Statistics`
- `SQLServer:General Statistics`
- `APP_POOL_WAS`
- `Web Service`
- `W3SVC_W3WP`
- `ASP.NET`
- `ASP.NET Applications`
