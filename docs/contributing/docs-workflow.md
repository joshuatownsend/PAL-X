# Docs workflow

This documentation site is generated by DocFX from the Markdown files under `docs/`. The toolchain is .NET-only — no Node, no JS, no `npm install`.

## Install DocFX

DocFX is pinned via `.config/dotnet-tools.json` at the repo root. To install:

```bash
dotnet tool restore
```

This installs the version the repo expects (2.78.5 today). The tool lives in `~/.dotnet/tools/` and is invoked as `dotnet docfx` or `docfx` if you have the tools directory on PATH.

## Preview locally

```bash
docfx docs/docfx.json --serve --port 8081
```

Serves at `http://localhost:8081/`. The port is set to 8081 deliberately — the default `8080` collides with the API in `Development` mode.

Edit any Markdown file under `docs/` and refresh the browser. DocFX rebuilds incrementally on file changes; no need to restart the server.

For a one-shot build (no server, just write `_site/`):

```bash
docfx docs/docfx.json
```

For a build that fails on warnings (matches CI):

```bash
docfx docs/docfx.json --warningsAsErrors
```

**Always run this before pushing a docs change.** It catches broken cross-references that DocFX would otherwise log as warnings and silently ship. The CI workflow uses this flag, so a warning is a CI failure waiting to happen.

## Editing an existing page

1. Find the page under `docs/<section>/<page>.md`.
2. Edit. Save.
3. Refresh the local preview.
4. Verify any links you added or changed resolve.
5. Run `docfx docs/docfx.json --warningsAsErrors` to confirm.
6. Commit.

## Adding a new page

1. Pick the section it belongs in: `concepts/`, `guides/`, `reference/`, `operations/`, `architecture/`, `contributing/`. If none fits, ask before creating a new section.
2. Create `docs/<section>/<your-page>.md` with front matter:

   ```markdown
   ---
   title: Your page title
   description: One-line description (≤160 chars for search).
   ---

   # Your page title
   ```

   The title in front matter and the H1 should match.

3. Add the page to `docs/<section>/toc.yml`:

   ```yaml
   - name: Your page title
     href: your-page.md
   ```

   (Or nest it under an existing group's `items:` array if there is one.)

4. Cross-reference it from related pages. The bidirectional link discipline is what makes the site navigable.

5. Run `docfx docs/docfx.json --warningsAsErrors` and verify clean.

## Front matter — what's required

Per the page authoring contract in our planning notes:

| Field | Required | Notes |
|---|---|---|
| `title` | yes | Matches the H1 below. |
| `description` | yes | ≤160 chars. Surfaces in search results and OG cards. |
| `uid` | no | DocFX cross-reference ID. Not used today; reserved. |

**Quote any description containing a `:` followed by a space.** YAML treats `key: value` as a mapping start; an unquoted `window: block` in the description triggers an "invalid YAML header" warning at build time. We've been bitten by this — quote defensively.

## Link conventions

Markdown internal links use relative paths:

```markdown
[Pack schema v1](../reference/pack-schema-v1.md)
[Within section](other-page.md)
[Subsection anchor](other-page.md#section-heading)
```

DocFX resolves these into `.html` paths at build time. If you write `.html` directly, the link breaks during local preview (Markdown files reference `.md` files).

For anchor links, DocFX generates the slug by lowercasing, replacing non-alphanumerics with `-`, and collapsing multiple `-`. For a heading `## PUT /api/orgs/{orgId}/members`, the slug is `put-api-orgsorgidmembers` (braces stripped, slashes become hyphens, hyphens collapsed). **When in doubt, link to the page without an anchor** — DocFX's slug rules are easy to get wrong.

## TOC files

Each section has a `toc.yml`. Three patterns:

### Flat — every page at the same level

```yaml
- name: Overview
  href: index.md
- name: First page
  href: first.md
- name: Second page
  href: second.md
```

### Grouped — `items:` nests pages under a label

```yaml
- name: Overview
  href: index.md
- name: Capture and analyze
  items:
    - name: Analyze a CSV
      href: analyze-csv.md
    - name: Analyze a BLG on Windows
      href: analyze-blg-windows.md
```

### Section homepage

```yaml
- name: ADRs
  href: adr/                    # Path to subdirectory
  homepage: adr/index.md         # The page rendered when ADRs is clicked
```

Use the section homepage form when a sub-section has its own index page.

## Diagrams

ASCII art works everywhere — GitHub Markdown, DocFX HTML, terminal `cat`, PR descriptions. Prefer it for system / flow diagrams. The data-flow page uses ASCII pipeline art for exactly this reason.

If you need a rendered diagram, Mermaid is the only option (DocFX supports it via fenced blocks). Mermaid only renders in HTML output; raw Markdown viewers show the source. Use sparingly.

## Images

Drop images under `docs/images/`. Reference with relative paths:

```markdown
![Diagram](../images/data-flow.png)
```

Resize before committing — the build serves images as-is, so a 5MB PNG bloats the site.

## When the build is unhappy

DocFX warnings break the CI build (`--warningsAsErrors`). The common ones:

| Warning | Cause | Fix |
|---|---|---|
| `InvalidFileLink` | Linked file doesn't exist | Check the relative path; check spelling |
| `InvalidBookmark` | Anchor doesn't match a heading | Read the slug generation rules; or drop the anchor |
| `invalid-yaml-header` | Front matter parsing error | Quote any description string containing a colon-space |

Run `docfx docs/docfx.json --warningsAsErrors` locally and read the first warning. The warning's source position tells you exactly where to look.

## What ships and what doesn't

The DocFX config (`docs/docfx.json`) excludes:

- `docs/internal/**` — planning, seed specs, legacy analysis. Reference for contributors; not for public docs.
- `docs/_site/` — the build output. Gitignored.
- `docs/reference/api/` — the auto-generated .NET API metadata (Step 10 feature).

If you put a Markdown file under `docs/internal/`, it won't appear on the public site.

## After merge

GitHub Actions on `main`:

1. Builds the site with `--warningsAsErrors`.
2. Uploads `_site/` as a Pages artifact.
3. Deploys via `actions/deploy-pages@v4`.

The site updates within ~2 minutes of a `main` merge. Pages must be enabled (`Settings → Pages → Source: GitHub Actions`).
